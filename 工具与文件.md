## 工具详解

### 1. opencode_sync_export

导出当前工作区的完整状态快照为 JSON 文件。

**收集内容：**
- opencode 配置文件（`~/.config/opencode/opencode.jsonc`）— 不含 secrets
- 环境变量名称（从 `opencode-dotfiles/.env` 提取，不含值）
- 所有子模块的路径、GitHub URL、当前 commit hash
- 已安装的 skills 列表（`~/.agents/skills/`）
- 平台和主机名元数据

**输出格式：**
```json
{
  "timestamp": "2026-06-11T...",
  "platform": "windows",
  "hostname": "my-machine",
  "opencodeConfig": { ... },
  "envVars": ["NOTION_TOKEN", "DEEPSEEK_API_KEY", ...],
  "submodules": [
    { "name": "my-project", "path": "projects/my-project", "url": "https://...", "commit": "a1b2c3d..." }
  ],
  "skills": ["alphafold-database", "gstack", ...]
}
```

### 2. opencode_sync_import

从 JSON 文件（由 export 生成）恢复工作区状态。

**恢复步骤：**
1. Clone 缺失的子模块 (`git clone <url> <path>`)
2. Fetch + reset 每个子模块到记录的 commit (`git fetch origin && git reset --hard <commit>`)
3. 合并 opencode config（deep-merge，不覆盖本地已有设置）
4. 从模板创建 `.env`（如不存在）

**dryRun 模式：** 只显示将要执行的变更，不实际修改。

### 3. opencode_sync_diff

对比当前状态与已保存的 JSON 文件。

**对比内容：**
- 子模块：commit 是否一致（显示 `→` 差异）
- Skills：是否有缺失的 skill

输出格式：`~ Submodule my-project: a1b2c3d → e4f5g6h`

### 4. opencode_sync_push

一键导出并推送。执行流程：

1. `exportSystemState()` → 写入 `workspace-sync-state.json`
2. `git add workspace-sync-state.json`
3. `git commit -m "<message>"`
4. `git push`

### 5. opencode_sync_pull

一键拉取并应用。执行流程：

1. `git pull`
2. 读取 `workspace-sync-state.json`
3. `importSystemState()` 恢复状态

**dryRun 模式：** 只显示状态元数据，不实际导入。

### 6. opencode_sync_status

列出所有子模块的详细状态。

**每个子模块显示：**
- ✅ 正常 — 目录存在、git 已初始化、无未提交更改
- ❌ 缺失 — 目录不存在
- ⚠️ 未初始化 — 目录存在但 `.git` 不存在
- 🔧 脏 — 有未提交更改

额外显示：当前 commit（前 7 位）、当前分支。

### 7. opencode_sync_verify

全面检查开发环境完整性的只读工具。

**检查项（7 项）：**

| 组件 | 验证方式 | 失败提示 |
|------|---------|---------|
| GitHub CLI | `gh auth status` | `gh auth login` |
| Git | `git --version` | 安装 Git |
| OpenCode config | 文件 `~/.config/opencode/opencode.jsonc` 存在 | 运行 setup |
| Ralph CLI | `ralph --version` | `npm install -g @wiggumdev/ralph` |
| Skills CLI | `skills --version` | `npm install -g skills` |
| Skills 目录 | `~/.agents/skills/` 目录 + 计数 | `npx skills add` |
| 子模块 | 逐个检测 missing/uninitialized/dirty | `git submodule update --init` |

### 8. opencode_sync_setup

一键初始化新设备工作区的破坏性工具。6 个可选开关：

| 选项 | 默认 | 功能 |
|------|------|------|
| `installGhCli` | true | 通过 winget/brew/apt 安装 GitHub CLI |
| `fixWindowsPaths` | true | 修复 Windows NTFS 文件名问题 |
| `copyConfig` | true | 从 dotfiles 复制 opencode config |
| `installRalph` | true | 安装 Ralph CLI（如缺失） |
| `installSkillsCli` | true | 安装 Skills CLI（如缺失） |
| `installSkills` | [] | 安装指定的 skills 包 |

**执行步骤：**
1. 安装 GitHub CLI（如缺失）
2. `git submodule update --init --recursive`
3. 修复 Windows NTFS 路径（仅 Windows，已知问题子模块列表）
4. 复制 `opencode-dotfiles/config/opencode.jsonc` → `~/.config/opencode/opencode.jsonc`
5. 安装 Ralph CLI（`npm install -g @wiggumdev/ralph`）
6. 安装 Skills CLI（`npm install -g skills`）
7. 安装 skills 包（`npx skills add <pkg> -g -y`）

### 9. opencode_sync_init

检测当前工作区信息。**此工具应仅在会话开始时调用一次。**

**检测逻辑：**
- 从当前目录向上遍历，查找 `.gitmodules` 文件
- 找到则返回：工作区名称、根路径、GitHub 远程 URL、dotfiles 存在状态、MCP 配置状态

**未找到工作区时：**
- 返回创建引导（建议名称 `codelib-{用户名}`）
- 列出创建步骤：`git init` → 创建 dotfiles 子模块 → `create_repo` → `setup`

### 10. opencode_sync_create_repo

通过 GitHub CLI 创建私人仓库。

**行为：**
- 默认创建 **私人** 仓库 (`gh repo create --private`)
- 自动设置 `git remote add origin`
- 如仓库已存在但为公开 → 警告并给出 `gh repo edit --visibility private` 命令
- `checkOnly=true` 时只检查仓库状态不创建

### 11. opencode_sync_api_keys

管理 API 密钥配置。

**三种 action：**

| action | 功能 |
|--------|------|
| `detect` | 扫描 `.env` + opencode config，列出所有 `*API_KEY*`/`*TOKEN*`/`*SECRET*` 变量名 |
| `generate` | 创建 `opencode-dotfiles/API key.md`，自动填入检测到的密钥名+说明+占位符 |
| `add` | 向已有 API key 文件添加一个密钥项 |

**生成的 API key.md 包含：**
- 环境变量表格（变量名 / 值 / 说明）
- 自动识别密钥类型（GitHub/Notion/DeepSeek/DashScope/Anthropic...）
- 可选的 GitHub Token 注入

### 12. opencode_sync_guide

生成 `SYNC-GUIDE.md` 到工作区根目录。

**文件内容：**
- 生成时间 + 源主机信息
- 需要安装的 MCP 服务器列表（类型、命令、URL）
- 所有已安装 skills 清单
- 所有子模块（名称、URL、commit）
- 分步恢复指南（verify → setup → API keys → import → restart）

此文件随 `push` 一起提交，新设备 pull 后由 agent 自动读取执行恢复。

## 核心逻辑层 (sync.ts)

所有函数纯 TypeScript，零外部依赖（仅 `node:*` 内置模块）。

### 数据接口

```typescript
WorkspaceState        // 工作区完整快照
SubmoduleState        // 单个子模块状态
ImportResult          // 导入操作结果
SetupResult           // 配置步骤结果
VerifyResult          // 验证项结果
SubmoduleStatusItem   // 子模块详细状态
WorkspaceInfo         // 工作区检测信息
RepoCreateResult      // 仓库创建结果
ApiKeyInfo            // API key 信息
```

### 函数清单

| 函数 | 职责 |
|------|------|
| `getPlatform()` | 返回 `windows`/`macos`/`linux` |
| `resolveWorkspaceRoot()` | 解析工作区根目录（优先读缓存） |

### 缓存机制

`opencode-dotfiles/.sync-cache.json` 在首次 `init` 时自动创建，记录了：

- 工作区根目录路径
- 工作区名称
- GitHub 远程 URL
- dotfiles 路径
- 创建/验证时间戳

后续所有工具调用 `resolveWorkspaceRoot()` 直接读缓存，**不再重复扫描目录**。MCP 每次被调用都是新进程，但缓存文件持久化，实现了"一次查找、永久记住"。
| `exportSystemState(root)` | 收集全部配置 → `WorkspaceState` |
| `importSystemState(root, state)` | 恢复子模块/config/.env |
| `diffState(current, saved)` | 对比两个状态的差异行 |
| `getSubmoduleStatus(root)` | 解析 `.gitmodules` + 逐个检测 |
| `verifyEnvironment(root)` | 7 项环境检查 |
| `setupWorkspace(root, opts)` | 6 步初始化配置 |
| `detectWorkspaceInfo(cwd?)` | 检测工作区，找不到返回 null |
| `createGitHubRepo(root, opts)` | gh repo create --private |
| `detectApiKeys(root)` | 扫描所有 API key 名称 |
| `initApiKeyFile(root, opts)` | 生成 API key.md |
| `generateSyncGuide(root, state)` | 生成 SYNC-GUIDE.md |
| `deepMerge(target, source)` | 深度合并配置对象 |

## agents 编排的工作流

### 备份工作区（上传）

```
opencode_sync_init
  └─ 检测工作区 → 若不存在，引导创建 codelib-{用户名}

opencode_sync_create_repo
  └─ 创建私人 GitHub 仓库，设置 remote

opencode_sync_api_keys (action=generate)
  └─ 生成 opencode-dotfiles/API key.md 模板

opencode_sync_setup
  └─ 安装 gh/git/submodules/config/ralph/skills

opencode_sync_export
  └─ 导出 workspace-sync-state.json

opencode_sync_guide
  └─ 生成 SYNC-GUIDE.md

opencode_sync_push
  └─ git add → commit → push
```

### 同步工作区（下载）

```
opencode_sync_pull
  └─ git clone + git pull，获取 workspace-sync-state.json + SYNC-GUIDE.md

(opencode_sync_diff)
  └─ [可选] 预览将要变更的内容

opencode_sync_setup
  └─ 按 SYNC-GUIDE.md 安装所有依赖

opencode_sync_api_keys (action=detect)
  └─ 列出需要填入的密钥

opencode_sync_import
  └─ 恢复子模块 + config

opencode_sync_verify
  └─ 验证环境完整性
```

### 日常使用

```
opencode_sync_status   → 随时查看子模块状态
opencode_sync_verify   → 随时检查环境健康
opencode_sync_push     → 工作告一段落，推送状态快照
```
